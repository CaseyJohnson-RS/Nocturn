## 1. Политика паролей

| Параметр              | Значение                            | Обоснование                                                                                                                         |
| --------------------- | ----------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Минимальная длина     | 8                                   | OWASP рекомендация; баланс между безопасностью и удобством                                                                          |
| Требования к символам | ≥1 заглавная, ≥1 строчная, ≥1 цифра | NIST SP 800-63B: complexity rules без спецсимволов - спецсимволы снижают usability без значимого прироста безопасности при длине ≥8 |
| Максимальная длина    | 128                                 | Защита от DoS при хешировании (Argon2id не имеет ограничения на длину входа, но 128 символов - разумный потолок для UI)             |

## 4. Политика тегов

| Параметр                         | Значение                                 | Обоснование                                                                               |
| -------------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------------------- |
| Минимальная длина имени          | 1                                        | Допускает короткие теги вроде «ЦТ», «AI»                                                  |
| Максимальная длина имени         | 50                                       | Достаточно для описательных тегов; не ломает UI                                           |

## 5. Лимиты и сроки

### 5.1 Сессии и токены

| Параметр                                 | Значение | Обоснование                                                                                                             |
| ---------------------------------------- | -------- | ----------------------------------------------------------------------------------------------------------------------- |
| Время жизни access-токена                | 15 мин   | Стандартный TTL; баланс между безопасностью и частотой refresh-запросов                                                 |
| Время жизни refresh-токена               | 14 дней  | 2 недели - стандартный баланс между удобством и безопасностью для персонального сервиса                                 |
| Максимум активных сессий на пользователя | 5        | Телефон + ПК + планшет + запас; при 25 пользователях - до 125 сессий, минимальная нагрузка                              |
| TTL ссылки подтверждения email           | 24 ч     | Достаточно чтобы пользователь проверил почту; не слишком долго для безопасности                                         |
| TTL ссылки сброса пароля                 | 1 ч      | OWASP рекомендация; сброс - операция, которую делают сразу                                                              |
| Срок удаления неподтверждённых аккаунтов | 72 ч     | 3× TTL подтверждения; даёт время на повторную отправку (SRS FR-AUTH-08)                                                 |
| Интервал heartbeat presence              | 10 сек   | Обнаружение отключения клиента; при закрытии вкладки или потере сети сервер обнаруживает broken pipe при следующем ping |
| TTL presence-ключа в Redis               | 30 сек   | Страховка на случай краша backend: если сервер не успел явно удалить ключ, он истечёт сам                               |

### 5.2 Заметки

| Параметр                                  | Значение | Обоснование                                                                                          |
| ----------------------------------------- | -------- | ---------------------------------------------------------------------------------------------------- |
| Максимум заметок на пользователя (включая soft-deleted) | 3 000    | SRS п.3.3: «до нескольких тысяч»; ожидание ~1500 → лимит с 2× запасом; 25 × 3000 = 75К заметок макс. |
| Максимальный размер заметки (символы)     | 20 000   | ~10 страниц текста;                                                                                  |
| Максимальная длина заголовка (символы)    | 200      | Помещается в UI списка заметок; достаточно для описательных заголовков                               |
| Максимум тегов на заметку                 | 10       | Категоризация без злоупотребления; больше 10 - признак отсутствия системы тегирования                |
| Срок хранения в корзине                   | 30 дней  | Стандартная практика (Gmail, Google Drive); SRS FR-NOTES-05d                                         |
| Интервал автосохранения                   | 5 сек    | Баланс между потерей данных и нагрузкой на сервер; при 25 пользователях - до 5 req/sec               |

### 5.3 AI-ассистент

| Параметр                                            | Значение | Обоснование                                                                           |
| --------------------------------------------------- | -------- | ------------------------------------------------------------------------------------- |
| Максимальная длина сообщения пользователя (символы) | 4 000    | ~3000 токенов при коэффициенте 1.3; достаточно для развёрнутого запроса; не перегружает контекст LLM       |
| Максимум источников в ответе                        | 5        | Больше 5 ссылок снижает usability; RAG-системы обычно используют top-3..5             |
| Максимум чат-сессий на пользователя                 | 50       | SRS FR-AI-14; при 25 пользователях - до 1250 сессий; минимальная нагрузка на БД       |
| Срок хранения чат-сессии                            | 14 дней  | Вполне достаточно; SRS FR-AI-07                                                       |
| Таймаут до первого токена LLM                       | 10 сек   | Максимально допустимое ожидание для UX; при превышении - запрос прерывается           |
| Максимум заметок в bulk-операции                    | 25       | SRS FR-AI-13; ограничивает нагрузку на proposals-генерацию; интуитивно понятный лимит |
| Макс. прикреплённых заметок к сообщению             | 5        | SRS FR-AI-11; достаточно для контекста; не перегружает токен-бюджет LLM               |
| Максимум сообщений в контексте LLM                  | 25       | Ограничивает размер контекста; достаточно для поддержания связного диалога            |
| Токены системного промпта (аппроксимация)           | 500      | Резерв для системного промпта при расчёте токен-бюджета                               |
| Safety margin (токены)                              | 500      | Компенсация погрешности аппроксимации символов→токены                                 |

## 6. Rate limiting

| Действие                                                | Лимит       | Обоснование                                                                                        |
| ------------------------------------------------------- | ----------- | -------------------------------------------------------------------------------------------------- |
| AUTH-эндпоинты (на IP)                                  | 10 req/мин  | Защита от brute-force; при 25 пользователях легитимный трафик ≪ 10 req/мин                         |
| Сброс пароля / повторная отправка подтверждения (на IP) | 3 req/мин   | Агрессивный лимит - операции с email не должны быть частыми                                        |
| Подтверждение email / применение ссылки сброса (на IP)  | 5 req/мин   | Одноразовые ссылки (SRS: «каждая ссылка одноразовая»); 5 - защита от перебора                      |
| Обновление токена (на IP)                               | 30 req/мин  | Refresh происходит каждые ~15 мин на сессию; 5 сессий = ~1.7/мин нормально; 30 - с большим запасом |
| CRUD-эндпоинты (на пользователя)                        | 120 req/мин | Автосохранение каждые 5 сек = 12 req/мин; + листинг, теги, корзина; 120 - с 5× запасом             |
| AI-эндпоинты (на пользователя)                          | 10 req/мин  | LLM-запросы дорогие; 10/мин - щедро для одного пользователя                                        |

## 7. Производительность

| Параметр                                    | Значение | Обоснование                                                                             |
| ------------------------------------------- | -------- | --------------------------------------------------------------------------------------- |
| Максимальное время загрузки страницы        | 3 сек    | Google Core Web Vitals: LCP ≤ 2.5 с (good); 3 с - допустимо для инструмента, не витрины |
| Максимальное время ответа CRUD API          | 500 мс   | Стандартный SLA для CRUD; воспринимается как «мгновенный» отклик                        |
| Максимальное время внутреннего поиска       | 1 сек    | Семантический поиск + полнотекстовый; при 75К заметок макс. - 1 с реалистично           |
| Таймаут первого токена LLM (перцентиль 95%) | 10 сек   | Совпадает с таймаутом в п.5.3; превышение = ошибка для пользователя                     |
